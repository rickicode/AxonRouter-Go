# Auth Specification

> Generated by openlore v1.0.0 on 2026-07-24 01:29

## Purpose

Orchestrates OAuth 2.0 token lifecycle across multiple providers, including authorization flow
initiation, token refresh with singleflight deduplication, connection-scoped persistence via CAS
pattern, and stale-caller rotation redirect through a time-bound cache.

## Requirements

> `internal/adminapi/key_manager.go`

The system SHALL initializes the keymanager and ensures a valid admin api key exists by loading from database or generating a new one.

#### Scenario: FreshInitializationCreatesNewKey
<!-- openlore-test: tags=smoke (auto) -->

- **GIVEN** Existing admin API key exists in database settings
- **WHEN** NewKeyManager is called
- **THEN** KeyManager holds the existing key value

The system SHALL returns the current master admin api key from memory without database lookup.

#### Scenario: ReadCurrentKeyFromMemory
- **GIVEN** KeyManager has been initialized with a valid key
- **WHEN** Current is called
- **THEN** The current admin API key string is returned
- **AND** No database access occurs

#### Scenario: ReadEmptyKeyBeforeInitializationCompletes
- **GIVEN** KeyManager has not been fully initialized
- **WHEN** Current is called
- **THEN** Empty string or current in-memory value is returned
- **AND** No database access occurs

### Requirement: RegenerateMasterKey

The system SHALL generates a new cryptographically secure admin api key, persists it to database settings, and updates the in-memory value.

#### Scenario: SuccessfulKeyRegeneration
<!-- openlore-test: tags=smoke (auto) -->
- **GIVEN** KeyManager is initialized with an existing key
- **WHEN** Regenerate is called
- **THEN** New key with axr_ prefix is returned and persisted
- **AND** New key is atomically updated in memory

#### Scenario: RegenerationFailsOnPersistence

### Requirement: RefreshAccessToken

The system SHALL the manager shall refresh access tokens using the provider's oauth service with singleflight deduplication.

<!-- openlore-test: priority=high (auto) -->
- **GIVEN** Credentials with an expired or expiring access token
- **WHEN** RefreshToken is called
- **THEN** New Credentials with updated access token are returned
- **AND** Uses singleflight.Group to prevent duplicate refresh requests
- **AND** Checks token expiration with 30-second buffer

### Requirement: RefreshTokenForConnection

The system SHALL the manager shall refresh tokens for a specific connection with stale-caller rotation redirect support.

#### Scenario: ConnectionTokenRefreshWithRotationHandling
<!-- openlore-test: priority=high (auto) -->
- **GIVEN** Connection-scoped credentials need refresh
- **WHEN** RefreshTokenForConnection is called
- **THEN** Fresh credentials are returned, with stale callers redirected via rotation cache
- **AND** Uses CAS pattern via TokenWriter for persistence
- **AND** Checks rotation cache for recently rotated tokens

### Requirement: RecordTokenRotation

The system SHALL the manager shall cache token rotation results to redirect stale callers.

#### Scenario: RotationCaching
<!-- openlore-test: priority=high (auto) -->
- **GIVEN** A successful token refresh produced new refresh token different from old
- **WHEN** recordRotation is called
- **THEN** The old-to-new token mapping is cached with 60-second TTL
- **AND** Entry TTL is 60 seconds
- **AND** Key format: provider:sha256(oldRefreshToken)

### Requirement: LookupTokenRotation

The system SHALL the manager shall lookup recent token rotations to redirect stale callers.

#### Scenario: RotationLookup
<!-- openlore-test: tags=smoke (auto) -->
- **GIVEN** A stale refresh token may have been recently rotated
- **WHEN** lookupRotation is called
- **THEN** The current valid credentials are returned if rotation occurred within TTL
- **AND** Returns nil if entry expired or not found
- **AND** Cleans up expired entries on lookup

### Sub-component: PersistTokens

Atomically persists token credentials and metadata for a connection with optimistic concurrency control.

#### Requirement: AtomicTokenPersistence

The system SHALL the persist tokens operation shall atomically store new access and refresh tokens for a connection using compare-and-swap semantics to prevent lost updates.

#### Scenario: SuccessfulCASUpdate
<!-- openlore-test: priority=high tags=smoke (auto) -->

- **WHEN** writer.SaveTokens is called with matching current version
- **THEN** the operation SHALL persist the tokens and return nil
- **AND** the stored record SHALL include the provider-specific metadata

#### Scenario: CasConflict
<!-- openlore-test: tags=regression (auto) -->

- **WHEN** writer.SaveTokens is called with stale version information
- **THEN** the operation SHALL return a compare-and-swap conflict error without updating
- **AND** the error SHALL indicate version conflict

#### Scenario: RotationRecorded

'new_access_456'
- **WHEN** recordRotation is called with the old token and new credentials
- **THEN** the operation SHALL store the rotation entry with current timestamp and configured TTL
- **AND** the cached entry SHALL be discoverable via lookupRotation for the TTL duration

#### Scenario: LookupReturnsNewCredentials
<!-- openlore-test: priority=high (auto) -->

- **WHEN** lookupRotation is called within 5 minutes of recording
- **THEN** lookupRotation for the old token SHALL return the rotationEntry containing 'new_access_456'
- **AND** subsequent lookupRotation calls SHALL return the new credentials within TTL
