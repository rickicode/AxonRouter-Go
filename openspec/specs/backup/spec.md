# Backup Specification

> Generated by openlore v1.0.0 on 2026-07-24 01:29

## Purpose

Handles HTTP requests for database backup download and restore operations, including
password-protected backups, multipart file uploads, and post-restore restart coordination.

## Requirements

> `internal/backup/restore.go`

### Requirement: Restore

The system SHALL reads an ndjson backup stream, validates its header, runs migrations, and restores all rows into the application database with retry logic for lock contention.

#### Scenario: SuccessfulFullRestore
<!-- openlore-test: tags=smoke (auto) -->
- **GIVEN** A valid NDJSON backup stream with a recognized header and matching schema version
- **WHEN** Restore is called
- **THEN** All rows are inserted into their respective tables and a summary result is returned
- **AND** Migrations are applied successfully
- **AND** No retryable database errors occur

#### Scenario: RetryOnTransientDatabaseErrors
<!-- openlore-test: tags=regression (auto) -->
- **GIVEN** A valid backup stream and a database under light contention
- **WHEN** Restore encounters a retryable error during row insertion
- **THEN** The operation retries with backoff and eventually succeeds
- **AND** The context is not cancelled
- **AND** The error matches retryable patterns (busy/locked/timeout)

#### Scenario: RejectInvalidHeaderVersion
<!-- openlore-test: tags=regression (auto) -->
- **GIVEN** A backup stream with an unsupported or mismatched header version
- **WHEN** Restore is called
- **THEN** Restore returns an error before modifying any data
- **AND** The header schema version does not match the application

### Requirement: RestoreRows

> Implementation: `Row` in `web/src/lib/components/ui/table/index.ts` · confidence: heuristic

The system SHALL iterates over the remaining ndjson lines, decrypts if needed, normalizes row data, and inserts batches into the target database with transaction retry logic.

#### Scenario: RestoreAllRowsFromBackup
- **GIVEN** A scanner positioned after the header with multiple valid NDJSON row lines
- **WHEN** restoreRows is called
- **THEN** All rows are persisted to their target tables and a summary with counts is returned
- **AND** Rows are grouped into batches and inserted via insertRestoreBatch
- **AND** Transactions are committed after all batches
- **AND** Retry logic handles transient failures

#### Scenario: RestoreEncryptedBackup
- **GIVEN** An encrypted backup stream and the correct password
- **WHEN** restoreRows is called with a password
- **THEN** Each line is decrypted before decoding and insertion
- **AND** Password is non-empty
- **AND** Lines are decrypted using DecryptLine

#### Scenario: ClearExistingDataBeforeRestore
- **GIVEN** A backup header specifying categories whose tables need clearing
- **WHEN** restoreRows processes the category list
- **THEN** Target tables are emptied in dependency-safe order before new rows are inserted
- **AND** DELETE statements are executed for tables in deleteOrder before inserts

### Requirement: DecodeRestoreRow

> Implementation: `Row` in `web/src/lib/components/ui/table/index.ts` · confidence: heuristic

The system SHALL decrypts a single backup line if encryption is enabled, then decodes the json into a row struct and normalizes its data values.

#### Scenario: DecodePlaintextRow
- **GIVEN** A plaintext NDJSON line representing a database row
- **WHEN** decodeRestoreRow is called with encrypted=false
- **THEN** A Row with normalized data is returned
- **AND** JSON numbers are preserved via UseNumber
- **AND** Map values are normalized via normalizeRowData

#### Scenario: DecodeEncryptedRow
- **GIVEN** An encrypted NDJSON line and the matching password
- **WHEN** decodeRestoreRow is called with encrypted=true
- **THEN** The line is decrypted and then decoded into a normalized Row
- **AND** DecryptLine successfully decrypts the ciphertext

### Requirement: NormalizeRowData

> Implementation: `Row` in `web/src/lib/components/ui/table/index.ts` · confidence: heuristic

The system SHALL normalizes json-decoded values to ensure consistent types for database insertion, including handling json.number and nested maps.

#### Scenario: NormalizeHeterogeneousRowData
- **GIVEN** A row map containing mixed JSON types including numbers and nested objects
- **WHEN** normalizeRowData is called
- **THEN** All values are converted to insertion-safe Go types
- **AND** Nested maps are recursively normalized
- **AND** json.Number values are preserved as strings

### Requirement: DeleteOrder

The system SHALL returns the ordered list of categories for deletion, ensuring dependent tables are cleared before their dependencies.

#### Scenario: DependencySafeDeletionOrder
- **GIVEN** A list of categories from the backup header
- **WHEN** deleteOrder is called
- **THEN** Categories are returned in an order that prevents constraint violations during DELETE
- **AND** The order respects foreign key constraints

### Requirement: InsertRestoreBatch

The system SHALL constructs and executes a single insert statement for a batch of rows into the specified table within a transaction.

#### Scenario: InsertBatchOfRows
- **GIVEN** A transaction and a batch of rows with uniform columns for a single table
- **WHEN** insertRestoreBatch is called
- **THEN** All rows are inserted in one statement and the transaction remains open
- **AND** Column names are quoted via quoteIdentifier
- **AND** Placeholders are generated for all values

#### Scenario: HandleVaryingColumnSets
<!-- openlore-test: tags=regression (auto) -->
- **GIVEN** Rows with map[string]any data where keys may vary
- **WHEN** insertRestoreBatch processes heterogeneous rows
- **THEN** The INSERT uses the union of all columns with NULLs for missing values
- **AND** Row values are flattened in column-sorted order

### Requirement: BeginRestoreTx

The system SHALL begins a database transaction for restore operations, retrying on transient lock/busy errors.

#### Scenario: BeginTransactionWithRetry
<!-- openlore-test: tags=smoke (auto) -->
- **GIVEN** A database under light contention
- **WHEN** beginRestoreTx is called
- **THEN** A valid transaction is returned after at most a few retries
- **AND** execWithRetry is used with isRetryableDBError

### Requirement: ExecWithRetry

The system SHALL executes an operation with exponential backoff retry when the database returns busy, locked, or timeout errors.

#### Scenario: RetrySucceedsOnTransientError
<!-- openlore-test: tags=regression (auto) -->
- **GIVEN** An operation that fails with SQLITE_BUSY on first attempt
- **WHEN** execWithRetry is called
- **THEN** The operation is retried and succeeds on a subsequent attempt
- **AND** The context is not cancelled
- **AND** The error matches retryable patterns

#### Scenario: FailOnNonRetryableError
<!-- openlore-test: tags=regression (auto) -->
- **GIVEN** An operation that fails with a permanent error
- **WHEN** execWithRetry is called
- **THEN** The error is returned immediately without retry
- **AND** The context is cancelled or the error is not retryable

### Requirement: IsRetryableDBError

> Implementation: `error` in `web/src/lib/stores.ts` · confidence: heuristic

The system SHALL determines whether a database error indicates a transient condition suitable for retry.

#### Scenario: DetectRetryableError
<!-- openlore-test: tags=regression (auto) -->
- **GIVEN** A database error with message 'database is locked'
- **WHEN** isRetryableDBError is called
- **THEN** true is returned
- **AND** Error message contains 'busy', 'locked', 'timeout', or 'database is locked'

#### Scenario: DetectNonRetryableError
<!-- openlore-test: tags=regression (auto) -->
- **GIVEN** A database error with message 'UNIQUE constraint failed'
- **WHEN** isRetryableDBError is called
- **THEN** false is returned
- **AND** Error message indicates a permanent failure like constraint violation

### Sub-component: RowRestoration

Coordinates the core restore workflow: scanning, decryption, transaction management, ordered deletion, batched insertion, and result aggregation.

- **AND** deletion order respects foreign key dependencies via deleteOrder

#### Scenario: ExhaustedRetries
<!-- openlore-test: tags=regression (auto) -->

- **THEN** the function SHALL return a non-nil error after max retries

#### Scenario: CancelledContext

- **AND** the error wraps context.Canceled or context.DeadlineExceeded
