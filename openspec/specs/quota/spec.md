# Quota Specification

> Generated by openlore v1.0.0 on 2026-07-24 01:29

## Purpose

Exposes administrative HTTP endpoints for quota management including listing quotas with filtering,
retrieving quota summaries with usage and reset information, and triggering quota refreshes for
specific connections.

## Requirements

> `internal/api/handlers/admin/quota.go`

### Requirement: List

> Implementation: `List` in `web/src/lib/components/ui/command/index.ts` · confidence: llm

The system SHALL the system shall list quotas with optional filtering by provider, search term, status, and pagination.

#### Scenario: ListQuotasWithDefaultPagination
- **GIVEN** Multiple quota records exist in the database
- **WHEN** List is called with no query parameters
- **THEN** Returns page 1 with up to 50 quota records
- **AND** Response contains paginated results
- **AND** Default pagination applied when not specified

#### Scenario: ListQuotasFilteredByProvider
- **GIVEN** Quota records exist for multiple providers
- **WHEN** List is called with provider query parameter
- **THEN** Returns only quotas matching the provider filter
- **AND** Only quotas for specified provider returned
- **AND** Pagination metadata reflects filtered result set

#### Scenario: ListQuotasWithSearchTerm
- **GIVEN** Quota records with searchable fields
- **WHEN** List is called with search query parameter
- **THEN** Returns quotas matching the search criteria
- **AND** Search matches against relevant quota fields
- **AND** Partial matches are supported

#### Scenario: ListQuotasFilteredByStatus
- **GIVEN** Quota records with varying statuses
- **WHEN** List is called with status query parameter
- **THEN** Returns only quotas with matching status
- **AND** Only quotas with specified status returned
- **AND** Status filter is case-sensitive or normalized as per domain rules

- **THEN** Returns comprehensive quota summary with resets and costs
- **AND** Each provider includes next reset timestamp

- **AND** Earliest reset is determined by comparing all provider reset times

### Sub-component: EarliestResetDetermination

> Implements: `earliestReset`

Identifies the nearest upcoming quota reset time across all providers from a map of reset timestamps.

#### Requirement: FindMinimumResetTime

The system SHALL the earliestresetdetermination operation shall identify the chronologically first reset timestamp from a collection of provider reset times, returning an empty string if no valid resets exist.

#### Scenario: MultipleValidTimestamps
- **GIVEN** resets map {'prov_a': '2024-02-01T00:00:00Z', 'prov_b': '2024-01-15T00:00:00Z', 'prov_c':
'2024-03-01T00:00:00Z'}
- **WHEN** earliestReset is called with the resets map
- **THEN** the operation SHALL return the chronologically earliest timestamp
- **AND** the returned string equals '2024-01-15T00:00:00Z'

#### Scenario: EmptyResetMap
- **GIVEN** an empty resets map
- **WHEN** earliestReset is called with no entries
- **THEN** the operation SHALL return an empty string
- **AND** the returned string is empty

#### Scenario: MixedValidInvalidDates
<!-- openlore-test: tags=smoke,regression (auto) -->
- **GIVEN** resets map {'prov_a': '2024-01-01T00:00:00Z', 'prov_b': 'not-a-date', 'prov_c': ''}
- **WHEN** earliestReset is called with partially invalid data
- **THEN** the operation SHALL ignore unparseable timestamps and return the earliest valid one
- **AND** the returned string equals '2024-01-01T00:00:00Z'
- **AND** invalid entries are skipped

> Implements: `Close`

#### Requirement: CloseDatabaseStore

The system SHALL the storeconnectiontermination operation shall gracefully close the underlying database connection pool, rejecting new operations and waiting for in-flight transactions to complete.

#### Scenario: CleanClosureIdlePool
- **GIVEN** a Store with an active, healthy database connection pool and no in-flight queries
- **WHEN** Close is called on the Store
- **THEN** the operation SHALL return nil and fully release database resources
- **AND** subsequent Store method calls return 'sql: database is closed' errors
- **AND** all idle connections are removed from the pool

#### Scenario: ClosureWithActiveQueries
- **GIVEN** a Store with long-running queries in progress
- **WHEN** Close is called while queries are executing
- **THEN** the operation SHALL wait for graceful termination up to configured timeout
- **AND** the operation blocks until queries complete or timeout
- **AND** returns context deadline exceeded error if timeout
